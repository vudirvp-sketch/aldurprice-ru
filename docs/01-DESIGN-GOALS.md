# Цели и принципы проектирования

> Документ фиксирует, **что** мы строим, **зачем** и **с какими ограничениями**. Это базовый референс для всех архитектурных и технических решений — если решение противоречит принципам здесь, нужно либо изменить принцип (с обоснованием), либо изменить решение.

## 1. Контекст и мотивация

### 1.1. Проблема

В Path of Exile 2 (PoE2) существует игровая механика «рунные комбинации» (runeshape), представленная в лиге «Runes of Aldur». Игрок собирает руны из убитых монстров и комбинирует их в специальные предметы (ремнант-комбинации), которые можно продать другим игрокам через торговую биржу. Цены на эти комбинации динамически меняются в зависимости от спроса и предложения, и игроку нужно быстро понять, какие комбинации выгоднее собрать и выставить на продажу.

Существующие community-инструменты для отслеживания цен в PoE2 ориентированы преимущественно на англоязычный клиент. Игроки с русским клиентом вынуждены либо переключать язык клиента на английский, либо использовать неоптимальные обходные пути. Кроме того, многие существующие инструменты потребляют заметно CPU/RAM из-за неоптимальных алгоритмов захвата и распознавания, что особенно чувствительно для пользователей со слабыми ПК.

### 1.2. Решение

AldurPrice — десктопное приложение для Windows, которое работает как click-through оверлей поверх окна PoE2. Приложение:

1. Захватывает регион экрана с панелью рунных комбинаций (через `PrintWindow` или `Windows.Graphics.Capture`)
2. Распознаёт текст названий предметов с помощью OCR (Windows Media OCR как primary, Tesseract как fallback)
3. Переводит русские названия на английский (для поиска в API цен) через многоуровневую систему маппингов
4. Запрашивает рыночные цены с community-сервисов (poe2scout, poe.ninja) и кэширует их локально
5. Отображает цветные ценники рядом с каждой строкой предмета, обновляя их в реальном времени

### 1.3. Подход: clean-room reverse engineering

Проект создаётся методом clean-room reverse engineering:

- **Не используется чужой код.** Мы наблюдали, как подобные инструменты работают (публично наблюдаемое поведение: какие функции предоставляют, как выглядят, какие API используют), но реализация написана с нуля.
- **Используются только публичные API и данные.** poe2scout и poe.ninja имеют публичные REST API. Данные переводов из Exiled Exchange 2 и poe2db.tw — публично доступны.
- **Не используется датамайн памяти игры.** Только захват экрана и публичные API — это безопасно с точки зрения user agreement GGG.
- **Соблюдаем лицензии источников.** Exiled Exchange 2 имеет свою лицензию (копируется в `ocr/translations/LICENSE`), poe2db.tw — fan-wiki с публичными данными, Tesseract — Apache 2.0.

## 3. Принципы проектирования

### 3.1. Русский язык — first-class citizen

Не «RU как fallback», а «RU по умолчанию, EN как fallback». Все алгоритмы (OCR-постпроцессинг, парсер количества, переводчик, stemmer) изначально проектируются под кириллицу, а латиница добавляется как один из вариантов. UI-строки с первого дня в `.resx`, никаких захардкоженных английских в code-behind.

Следствие: тестируем на RU-данных в первую очередь, EN-данные — во вторую. Если алгоритм ломается на кириллице, это критический баг, а не «edge case».

### 3.2. Производительность — через избегание работы

Лучший способ сделать что-то быстро — не делать это. Применяем три принципа:

- **Skip when unchanged.** Frame-differ на perceptual hash пропускает OCR, если регион не изменился с прошлого кадра.
- **Skip when irrelevant.** Если foreground window ≠ PoE2 — никаких захватов, никаких OCR.
- **Skip when closed.** Детектор панели по яркости верхней строки пропускает весь пайплайн, если панель рунешейпов закрыта.

Дополнительно: адаптивный polling замедляет интервал при высокой CPU-нагрузке системы.

### 3.3. Persistence по умолчанию

Любое состояние, которое дорого пересчитать, должно persist'иться на диск:

- Кэш цен → SQLite (`prices.db`)
- Настройки → `config/appsettings.json`
- Логи → `logs/app-YYYYMMDD.txt` (rolling)
- Crash-логи → `crash-reports/crash-*.txt`
- Debug images → `debug-images/*.png` (по требованию)

In-memory кэш — только как hot layer поверх persistent. При рестарте всё восстанавливается из persistence.

### 3.4. Слои без циклических зависимостей

Захват → OCR → Перевод → Цены → Оверлей — однонаправленный пайплайн. Каждый слой зависит только от контрактов (`Contracts/`) из предыдущего, никогда не от реализации. Это позволяет:

- Тестировать каждый слой изолированно (mock'и на контрактах)
- Заменять реализации (Windows OCR ↔ PaddleOCR в будущем)
- Компилировать слои параллельно (быстрее build)

### 3.5. DI-first, но без overengineering

Используем `Microsoft.Extensions.DependencyInjection` для всех сервисов с lifetime `Singleton` (stateful) или `Transient` (stateless). Не вводим фабрики фабрик, plugin-архитектуру, или динамическую загрузку модулей, если не нужно — YAGNI.

### 3.6. Потокобезопасность по умолчанию

Все shared-состояния (кэш цен, кэш переводов, recent translations) — `ConcurrentDictionary` или `lock`-защищённые. Background воркеры общаются с UI через `Dispatcher.Invoke` (WPF) или `IChannel<T>` для очередей сообщений. Никаких `async void` кроме event handlers.

### 3.7. Тестопригодность

Каждый публичный метод — чистая функция или сервис с inject'аемыми зависимостями. Никаких `DateTime.Now` напрямую (через `ISystemClock`), никаких `File.ReadAllText` напрямую (через `IFileSystem` абстракцию), никаких `HttpClient` напрямую (через `IHttpClientFactory`). Это позволяет тестам подменять любую зависимость.

### 3.8. Configuration через strongly-typed options

Все настройки — `IOptionsMonitor<T>` с POCO классами (`AppOptions`, `OcrOptions`, `PricingOptions`). Валидация через `IValidateOptions<T>`. JSON-схема для автокомплита в редакторах. Hot-reload через `FileSystemWatcher`. Никаких магических строк или `Configuration["Foo:Bar"]` в коде.

## 4. Что НЕ делаем

Зафиксируем явно, чтобы не размывать фокус:

- **Не поддерживаем Linux/macOS.** WPF — Windows-only. Кроссплатформенность через Avalonia — в бэклоге после v1.0.
- **Не вмешиваемся в память игры.** Только screen capture и публичные API. Это безопасно с точки зрения anti-cheat.
- **Не используем official trade API GGG.** Требует OAuth и rate limits. Используем community API (poe2scout, poe.ninja).
- **Не делаем биржу.** Только цены, без автоматической покупки/продажи.
- **Не делаем inventory overlay.** Только панель рунных комбинаций. Инвентарь — domain Exiled Exchange 2.
- **Не поддерживаем эксклюзивный fullscreen.** Только Borderless Windowed или Windowed — иначе не работает screen capture.
- **Не запускаемся без .NET 9 runtime.** Self-contained single-file, ~120 МБ.
- **Не делаем plugin-систему.** Все источники цен, OCR-движки, переводчики — захардкожены в DI. Если понадобится plugin — добавим после v1.0.

## 5. Нефункциональные ограничения

### 5.1. Технологический стек

| Компонент | Технология | Обоснование |
|---|---|---|
| Runtime | .NET 9 | Современный, performant, long-term support |
| UI | WPF | Векторная графика, MVVM, PerMonitorV2 DPI |
| DI / Hosting | Microsoft.Extensions.Hosting | Стандарт .NET ecosystem |
| OCR primary | Windows.Media.Ocr | Нативно в Windows 10 1809+, быстро, без зависимостей |
| OCR fallback | Tesseract 5.2 | Кроссплатформенный, Apache 2.0, поддержка 100+ языков |
| HTTP | HttpClient + IHttpClientFactory | Стандарт, pooling, retry policies |
| JSON | System.Text.Json + source generation | Быстро, без reflection, AOT-совместимо |
| Persistence | Microsoft.Data.Sqlite (SQLite + WAL) | Embedded, без сервера, ~1 МБ |
| Logging | Serilog | Structured logging, multiple sinks |
| MVVM | CommunityToolkit.Mvvm | Официальный toolkit, source generators |
| Установщик | Inno Setup 6 | Стандарт для Windows-приложений, бесплатный |
| CI | GitHub Actions | Бесплатный для open source |

### 5.2. Целевые платформы

- **Windows 11** — primary, тестируем на каждой сборке
- **Windows 10 build 1809+** — поддерживаем, тестируем перед релизом
- **Windows 7/8** — только Tesseract fallback (Windows OCR недоступен), best-effort без гарантий
- **Linux/macOS** — не поддерживаем до v2.0 (Avalonia migration)

### 5.3. Целевая аудитория

- **Основная:** русскоязычные игроки PoE2, использующие официальный русский клиент
- **Вторичная:** англоязычные игроки, которые хотят альтернативу существующим инструментам с фокусом на производительность
- **Не целевая:** casual-игроки, которые не интересуются экономикой; профессиональные трейдеры, которым нужна full-функциональность биржи

### 5.4. Лицензионные ограничения

- Все исходники — MIT, free for any use
- Данные Exiled Exchange 2 — под их лицензией (см. `ocr/translations/LICENSE`), проверять перед коммерческим использованием
- Tesseract traineddata — Apache 2.0
- poe2db.tw — fan-wiki, attribution required
- poe2scout/poe.ninja API — публичные, но с неявными rate limits (нужно уважать, не более 1 запроса в 15 минут)

## 6. Критерии успеха проекта

Проект считается успешным когда:

1. **v1.0.0 выпущен** с полной функциональностью (см. [05-ROADMAP.md](05-ROADMAP.md))
2. **Все метрики NF1–NF8** достигнуты (см. таблицу выше)
3. **100% рунных комбинаций** с poe2db.tw корректно переводятся в цены
4. **5+ тестировщиков** из сообщества подтвердили стабильность на их конфигурациях
5. **Документация** полная и понятная (новый разработчик может начать контрибьюшить за <1 часа)
6. **CI green** на всех платформах (Windows 10/11, разные DPI)

Проект НЕ считается успешным, если:

- Стабильность OCR <95% на RU-клиенте
- CPU >15% при активной панели
- Crash'и при нормальном использовании
- Локализация неполная (есть захардкоженные EN строки)
- Документация устарела или неполна
